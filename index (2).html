Step 1. 가중치 결정 (resolve_weights)
가중치는 아래 우선순위 순서로 결정돼요. 위에 있을수록 먼저 적용됩니다.
1
프로필 저장값 (user_weights)
회원가입 때 슬라이더로 설정하고 저장한 값. profile_service.json에 저장되어 있으면 항상 이게 먼저 적용돼요.
2
즉석 슬라이더 (w_price / w_protein / ...)
검색 화면에서 직접 조정한 값. 5개 합이 자동으로 1.0이 되도록 정규화됩니다.
3
모드 프리셋 (mode)
슬라이더 설정이 없을 때 사용. 가성비 / 고단백 / 저칼로리 / 저염 / 기본 중 선택.

Step 2. 정규화 (normalize_scores)
단위가 제각각인 5개 기준을 0~1 사이로 통일해요. 1,146개 레시피 전체에서 최댓값을 기준으로 나눕니다.
기준정규화 공식해석예시
가격1 − (가격 / 최대가격)싸면 높은 점수200원 → 약 0.95점
단백질단백질 / 최대단백질많을수록 높은 점수30g → 약 0.7점
칼로리1 − (칼로리 / 최대칼로리)낮을수록 높은 점수150kcal → 약 0.8점
나트륨1 − (나트륨 / 최대나트륨)낮을수록 높은 점수200mg → 약 0.85점
조리시간1 − (시간 / 최대시간)빠를수록 높은 점수15분 → 약 0.75점
Step 3. 기본 점수 산출 (calc_score)
BaseScore = w_price · s_price + w_protein · s_protein + w_calorie · s_calorie + w_sodium · s_sodium + w_time · s_time
구체 예시 — 가격 슬라이더를 80%로 설정했을 때
재료비 200원 레시피 → s_price ≈ 0.95 → 0.80 × 0.95 = 0.76점 (가격 기여분만)
재료비 8,000원 레시피 → s_price ≈ 0.11 → 0.80 × 0.11 = 0.09점
→ 가격 슬라이더를 높이면 저렴한 레시피가 훨씬 유리해짐
Step 4. 이력 기반 미세 조정 (blend_with_log)
최종 가중치 = 0.75 × 기본 가중치 + 0.25 × 이력 취향 신호
먹은 음식 이력이 3개 이상 쌓이면 작동해요.
사용자가 계속 저렴한 레시피를 골랐다면 → 가격 가중치가 소폭 자동 올라감
α=0.25로 설정해서 기본 설정이 너무 많이 바뀌지 않도록 제한.
Step 5. MMR 재정렬 — 다양성 확보
MMR(d_i) = 0.7 × Relevance(d_i) − 0.3 × max_sim(d_i, 선택된 레시피)
❌ 기존 방식 (교수님 지적)
유사도 0.8이면
점수 × (1 − 0.8×0.8) = 점수 × 0.36
→ 점수 64% 날아감
→ 관련 레시피가 순위 밖으로 밀림
✅ 현재 방식 (MMR)
점수는 그대로 유지
이미 뽑힌 것과 비슷하면
→ 순서만 뒤로 미룸
→ 다양한 레시피 노출
λ=0.7 의미: 관련성(70%)을 우선시하면서 다양성(30%)을 보조. 비율 조정 가능.

근본 문제: lprice를 단가로 착각
네이버 쇼핑 API의 lprice는 상품 한 개의 총 판매가예요. 이걸 100g 단가처럼 쓰면 완전히 틀린 값이 나옵니다.
❌ 잘못된 방식 (초기 코드)
"양파 3kg 9,900원" 검색
lprice = 9,900원 그대로 단가 사용
→ 양파 30g = 9,900원 × 0.3 = 2,970원
→ 말이 안 되는 금액
✅ 수정된 방식
상품명에서 "3kg" 추출
9,900 ÷ 3,000g × 100 = 330원/100g
→ 양파 30g = 99원
→ 현실적인 금액
3단계 가격 파이프라인
1
KAMIS API — 농산물유통공사 공식 시세
배추, 양파, 감자 등 신선 농산물과 수산물. 정부 공식 일별 도매가라 가장 신뢰도 높음.
2
네이버 쇼핑 API — 가공품·조미료 실거래가
된장, 고춧가루, 참기름 등 KAMIS에 없는 재료. 여러 상품의 중앙값을 사용해 이상값 영향을 줄임.
3
STANDARD_AVERAGE_DB — 직접 조사한 기준가
1, 2단계에서 못 찾은 재료의 최후 수단. 300여 품목을 직접 조사해서 DB에 넣어둔 값.

피드백 ③
